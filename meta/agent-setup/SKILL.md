# Agent Setup — Credential Collector

## Purpose

After the Agent Builder (requirement-gatherer + skill-factory) creates skills, this skill collects the required ENV variables from the user conversationally and writes them to `openclaw.json`. It also validates credentials where possible and generates `.env.example` files for portability.

## Triggers

- **Auto-trigger**: skill-factory completes and reports ENV vars needed
- **Manual trigger**: `/setup {skill-name}` to reconfigure a skill's credentials
- **Bulk trigger**: `/setup {org}` to check all skills under an org

## Flow

### Step 1: Identify Required ENV Vars

Read the skill(s) that need configuration. Sources:

**From skill-factory output:**
The skill-factory passes a list of skills just created with their ENV vars.

**From manual /setup:**
```bash
# Read the SKILL.md to find required env vars
cat {skill-path}/SKILL.md
```
Extract `metadata.openclaw.requires.env` array from SKILL.md frontmatter.

### Step 2: Check What's Already Configured

Read current `~/.openclaw/openclaw.json` and check `skills.entries.{skill-name}.env` for each required var.

Mark each var as:
- **configured** — value exists and is non-empty
- **missing** — not set or empty string

Report:
```
🔧 Setting up {skill-name}

{N} credentials needed, {M} already configured:
  ✓ JIRA_BASE_URL — already set
  ✗ JIRA_API_TOKEN — missing
  ✗ JIRA_USER — missing
```

### Step 3: Collect Missing Credentials

For each missing ENV var, ask the user ONE AT A TIME via chat:

```
{current}/{total}: {ENV_VAR_NAME}
→ {description from skill's env_vars}
→ {help_url — see validation-patterns.md for common services}

Paste the value here:
```

**Rules for collection:**
- Ask ONE var at a time — wait for response before asking next
- NEVER echo back the value after receiving it
- Confirm with just: `✓ {ENV_VAR_NAME} — saved`
- If user says "skip" → mark as skipped, continue to next
- If user says "cancel" → stop, report what was configured so far

### Step 4: Write to openclaw.json

After receiving each value, immediately write it to openclaw.json:

```json
{
  "skills": {
    "entries": {
      "{skill-name}": {
        "enabled": true,
        "env": {
          "{ENV_VAR_NAME}": "{user-provided-value}"
        }
      }
    }
  }
}
```

Read the current openclaw.json, update the specific skill's env block, write back.

**CRITICAL SECURITY RULES:**
- NEVER log the actual credential value anywhere
- NEVER echo the value back to the user
- NEVER include credentials in git commits
- NEVER write credentials to any file in the agent-skills repo
- Only write to `~/.openclaw/openclaw.json` (local, not in git)

### Step 5: Validate Credentials (Optional)

After all credentials are collected, attempt validation where possible.

Read `{baseDir}/references/validation-patterns.md` for service-specific test patterns.

Common validations:
- **API tokens**: Make a lightweight API call (e.g., GET /myself for Jira)
- **Webhooks**: Send a test message (with user permission)
- **Database URLs**: Attempt connection
- **SMTP**: Verify connection (don't send email)

Report:
```
Testing credentials...
  ✓ JIRA_API_TOKEN — Jira API responded (200 OK)
  ✓ JIRA_BASE_URL — valid Jira instance
  ⚠ SMTP_HOST — could not verify (will test on first use)
```

If validation fails:
```
  ✗ GITHUB_TOKEN — authentication failed (401)
  → Want to re-enter this value?
```

### Step 6: Generate .env.example

Create a `.env.example` file for the agent template (pushed to GitHub, NO actual values):

```bash
# Agent: {org}-{agent-id}
# Generated by Agent Builder on {date}
# Skill: {skill-name}

{ENV_VAR_1}=           # {description}
{ENV_VAR_2}=           # {description}
```

Save to: `agent-skills/orgs/{org}/agent-templates/{agent-id}/.env.example`

### Step 7: Generate openclaw-entries.json

Create a JSON snippet for portable agent config:

```json
{
  "agent": {
    "id": "{org}-{agent-id}",
    "workspace": "{workspace-path}",
    "model": "openai-codex/gpt-5.4"
  },
  "skills": {
    "{skill-name}": { "enabled": true, "env": {} }
  },
  "bindings": [],
  "cron": []
}
```

Save to: `agent-skills/orgs/{org}/agent-templates/{agent-id}/openclaw-entries.json`

This file has the structure but NO credential values — it's safe to push to GitHub.

### Step 8: Report Completion

```
✅ All {N} credentials configured for {skill-name}:
  ✓ {VAR_1} — saved {+ validated}
  ✓ {VAR_2} — saved {+ validated}
  ⚠ {VAR_3} — skipped

{if all validated:}
Skill {skill-name} is ready to use.
Trigger with: "{example trigger phrase}"

{if some skipped:}
{M} credentials were skipped. Run /setup {skill-name} to complete configuration.

{if agent template generated:}
Agent template saved to: agent-skills/orgs/{org}/agent-templates/{agent-id}/
Files: .env.example, openclaw-entries.json
```

## Reconfiguration

When user runs `/setup {skill-name}` on an already-configured skill:

1. Show current state (configured vs missing — NEVER show actual values)
2. Ask: "Which credential do you want to update? (all / specific var name)"
3. Collect and overwrite the specified values
4. Re-validate

## Bulk Setup

When user runs `/setup {org}`:

1. Find all skills under `orgs/{org}/skills/`
2. Check each skill's env status
3. Report summary: "3 skills configured, 2 need credentials"
4. Offer to configure missing ones sequentially

## Rules

- NEVER skip credential collection — always ask the user
- NEVER echo back credential values
- NEVER store credentials in git-tracked files
- NEVER include credentials in chat logs or memory files
- Always write to openclaw.json per-skill env blocks
- One credential at a time — don't overwhelm the user
- Validate when possible — catch typos early
- Generate .env.example for portability (no values, just var names)

## Error Handling

- openclaw.json read failure → warn user, provide manual instructions
- openclaw.json write failure → show the JSON snippet for manual application
- Validation timeout → mark as "unverified", continue
- User provides empty value → ask again or allow skip
- Skill not found → check spelling, list available skills
